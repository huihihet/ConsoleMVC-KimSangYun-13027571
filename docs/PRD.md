# PRD — ConsoleMVC PoC

## 1. 목적

MVC 아키텍처 패턴을 Java 콘솔 환경에서 패키지 구조로 검증한다.  
프레임워크 없이 순수 Java로 Model·Controller·View 역할 분리가 실제로 동작하는지 확인하고,  
SSemi 본 프로젝트에서 재사용할 MVC 패턴 기반을 수립한다.

---

## 2. 배경 및 문제 정의

| 현황 | 문제 |
|------|------|
| 단일 클래스에 I/O·로직·데이터 혼재 | 변경 범위 예측 불가, 테스트 작성 곤란 |
| 레이어 간 의존 방향 불명확 | 기능 추가 시 전파 범위 파악 어려움 |

**목표**: 각 레이어를 독립적으로 수정·테스트할 수 있는 구조를 PoC로 시연한다.

---

## 3. 범위 (Scope)

### In-Scope

- S-Semi 도메인의 **시료(Sample) 관리** CRUD 흐름
- Model / Controller / View 패키지 분리
- 인메모리 저장소 구현
- 각 레이어 단위 테스트

### Out-of-Scope

- DB·파일 영속성 (DataPersistence PoC 담당)
- 주문(Order) 처리 및 생산 라인 (SSemi 본 프로젝트 담당)
- 네트워크·API, 인증·권한, 멀티 스레딩

---

## 4. 도메인: 시료(Sample) 관리

### 엔티티

```
Sample {
    sampleId         : Long    // 자동 증가, 1부터 시작
    name             : String  // 시료 이름, 공백 불허
    avgProductionTime: int     // 평균 생산 시간(분), 1 이상
    yield            : double  // 수율 (0.0 초과 ~ 1.0 이하)
    stock            : int     // 현재 재고 수량, 0 이상
}
```

### 기능 요구사항

| ID | 기능 | 설명 |
|----|------|------|
| F-01 | 시료 등록 | 이름·평균생산시간·수율 입력 → 저장 (초기 재고 0) |
| F-02 | 시료 목록 조회 | 등록된 모든 시료를 ID 오름차순으로 표 출력 (재고 포함) |
| F-03 | ID 단건 조회 | ID로 시료 상세 조회, 없으면 오류 메시지 출력 |
| F-04 | 시료 정보 수정 | 이름·평균생산시간·수율·재고 변경 (없는 ID면 오류) |
| F-05 | 시료 삭제 | ID로 삭제 (없는 ID면 오류) |
| F-06 | 이름 검색 | 입력 문자열을 포함하는 시료 목록 출력 |
| F-07 | 메뉴 네비게이션 | 번호 입력으로 기능 선택, `0` 입력 시 종료 |

### 입출력 시나리오

```
=== S-Semi 시료 관리 시스템 ===
1. 시료 등록
2. 시료 목록 조회
3. 시료 조회 (ID)
4. 시료 수정
5. 시료 삭제
6. 이름 검색
0. 종료
선택 > _
```

**오류 메시지 규칙**

| 상황 | 메시지 |
|------|--------|
| 잘못된 메뉴 번호 | `"유효하지 않은 메뉴입니다."` → 재입력 |
| 존재하지 않는 ID | `"해당 시료를 찾을 수 없습니다."` |
| 수율 범위 초과 | `"수율은 0.0 초과 1.0 이하로 입력하세요."` |
| 생산 시간 오입력 | `"평균 생산 시간은 1 이상의 정수로 입력하세요."` |
| 재고 음수 입력 | `"재고 수량은 0 이상으로 입력하세요."` |

## 5. 비기능 요구사항

| 항목 | 기준 |
|------|------|
| 빌드 | `./gradlew build` 경고 없이 성공 |
| 테스트 커버리지 | Model·Controller 핵심 로직 80% 이상 |
| 의존 방향 | Controller → Model, Controller → View (역방향 금지) |
| 외부 의존성 | JUnit 외 추가 라이브러리 없음 |

## 6. 아키텍처 설계

### 레이어 다이어그램

```
┌──────────────────────────┐
│           Main           │  객체 조립 · 루프 실행
└────────────┬─────────────┘
             │ Constructor Injection
             ▼
┌──────────────────────────┐
│     SampleController     │  입력 수신 · 흐름 제어
└────────┬─────────────────┘
         │              │
         ▼              ▼
┌──────────────┐  ┌──────────────┐
│    Model     │  │     View     │
│  (Sample +   │  │  SampleView  │
│  Repository) │  │              │
└──────────────┘  └──────────────┘
```

### 클래스 책임 요약

| 클래스 | 패키지 | 책임 |
|--------|--------|------|
| `Sample` | `model.entity` | 시료 도메인 데이터 보유, 유효성 검증 |
| `SampleRepository` | `model.repository` | CRUD 계약 정의 (인터페이스) |
| `InMemorySampleRepository` | `model.repository` | 리스트 기반 인메모리 구현 |
| `SampleController` | `controller` | 입력 파싱, 모델 호출, 뷰 위임 |
| `SampleView` | `view` | 출력 포맷팅 및 `System.out` 렌더링 |
| `Router` | `app` | 메뉴 번호 → Controller 메서드 라우팅 |
| `Main` | (루트) | 객체 조립 및 루프 실행 |

## 7. 수용 기준 (Acceptance Criteria)

- [ ] `./gradlew build`가 경고 없이 성공한다
- [ ] F-01~F-07 기능이 콘솔에서 정상 동작한다
- [ ] Controller 클래스에 `System.out` 직접 호출이 없다
- [ ] Model 클래스에 `Scanner` 또는 콘솔 I/O 코드가 없다
- [ ] View 클래스에 비즈니스 로직(조건 분기·계산)이 없다
- [ ] 잘못된 입력 시 오류 메시지를 출력하고 재입력을 받는다
- [ ] `./gradlew test`에서 모든 테스트가 통과한다
- [ ] Model·Controller 커버리지 80% 이상
